# AI Feature Request: Lightweight UART Bootloader and Flashing Tool

## 1. Project Background

The previous project was based on an AUTOSAR-style toolchain where an AUTOSAR configuration tool was used together with multiple drivers to simulate UDS services and allow the bootloader to perform flashing operations.

However, that architecture makes the bootloader too heavy, difficult to maintain, and hard to port to different hardware platforms. The new project must redesign the system with a lightweight, portable, modular bootloader and a new flashing tool built from scratch.

The new architecture will be split into two main responsibilities:

1. **Bootloader AI**
   - Responsible for designing and implementing the embedded bootloader in C.
   - Must define the flashing protocol, frame format, bootloader state machine, and interface contract first.
   - Must generate clear documentation so that another AI can implement the PC flashing tool independently.

2. **Flashing Tool AI**
   - Responsible for implementing the PC-side flashing tool in Python with Qt6.
   - Must follow the protocol and flow generated by the Bootloader AI.
   - Must support firmware flashing through UART and optional firmware signing.

---

## 2. Main Goal

Create a lightweight, portable, safe, and maintainable bootloader system that can flash application firmware over UART using a custom protocol.

The system must be designed so that:

- The bootloader remains small and easy to port.
- Hardware-dependent code is isolated from protocol and application logic.
- The flashing protocol is clearly documented and stable.
- The PC flashing tool can be developed independently by another AI.
- Each development phase is split into milestones.
- Code is reviewed, built, and tested at each milestone before continuing.
- The AI must avoid generating large unverified code blocks without build or test validation.

# 5. Bootloader Requirements

## 5.1 General Requirements

The bootloader shall:

- Be written in C.
- Be lightweight and portable.
- Support flashing firmware over UART.
- Use a clearly defined custom frame format.
- Support checksum validation.
- Support optional firmware signature verification.
- Allow signature verification to be enabled or disabled by macro.
- Support configurable logging over UART or USB.
- Provide a clean module separation.
- Use a centralized configuration folder.
- Be easy to port to a new MCU or hardware platform.
- Follow clean code and MISRA C-oriented style.
- Refer to the existing codebase only for component structure and coding style, not for copying outdated logic blindly.

---

## 5.2 Bootloader Design Principles

The bootloader must follow these principles:

1. **Small core**
   - Keep the bootloader minimal.
   - Avoid unnecessary AUTOSAR-heavy abstractions.
   - Avoid complex dependencies.

2. **Portable hardware abstraction**
   - Hardware-dependent code must be isolated.
   - Porting to new hardware should mainly require changes in HAL/config files.

3. **Layered architecture**
   - Separate protocol, transport, flash, security, logging, and application jump logic.

4. **Protocol-first development**
   - The bootloader AI must define the flashing flow and frame format before implementing the full bootloader.

5. **Fail-safe communication**
   - Communication must include timeout handling, error response, retry support, and session reset.

6. **Recoverable flashing process**
   - If flashing fails, the bootloader must clear the current flashing session and be ready for a new flash attempt.

7. **Milestone-based implementation**
   - Do not implement everything in one large step.
   - Each milestone must be built, tested, reviewed, and committed.

## 9.2 Frame Format

The protocol should use a deterministic frame format.

Recommended frame format:

```text
+------------+------------+------------+------------+------------+---------------+------------+
| SOF        | Version    | Command ID | Sequence   | Length     | Payload       | Checksum   |
| 1 byte     | 1 byte     | 1 byte     | 1 byte     | 2 bytes    | N bytes       | 2 bytes    |
+------------+------------+------------+------------+------------+---------------+------------+
```

### Field Description

| Field | Size | Description |
|---|---:|---|
| SOF | 1 byte | Start of frame marker, recommended value `0xA5` |
| Version | 1 byte | Protocol version, initial value `0x01` |
| Command ID | 1 byte | Command or response identifier |
| Sequence | 1 byte | Incrementing sequence number used to detect duplicate or lost frames |
| Length | 2 bytes | Payload length in bytes, little-endian |
| Payload | N bytes | Command-specific payload |
| Checksum | 2 bytes | CRC16 or checksum over Version, Command ID, Sequence, Length, and Payload |

Recommended checksum:

```text
CRC16-CCITT-FALSE
Polynomial: 0x1021
Initial value: 0xFFFF
```

The Bootloader AI may choose another checksum only if justified clearly.

---

## 9.3 Command List

| Command | ID | Direction | Description |
|---|---:|---|---|
| HELLO | `0x01` | Tool → Bootloader | Start handshake and check bootloader availability |
| HELLO_RESPONSE | `0x81` | Bootloader → Tool | Return bootloader version, protocol version, and capability flags |
| START_SESSION | `0x02` | Tool → Bootloader | Start flashing session |
| START_SESSION_RESPONSE | `0x82` | Bootloader → Tool | Confirm session start |
| ERASE | `0x03` | Tool → Bootloader | Erase application flash area |
| ERASE_RESPONSE | `0x83` | Bootloader → Tool | Confirm erase result |
| DOWNLOAD_START | `0x04` | Tool → Bootloader | Send firmware metadata before transfer |
| DOWNLOAD_START_RESPONSE | `0x84` | Bootloader → Tool | Confirm metadata validation |
| DATA | `0x05` | Tool → Bootloader | Send firmware data block |
| DATA_RESPONSE | `0x85` | Bootloader → Tool | Confirm block write result |
| DOWNLOAD_END | `0x06` | Tool → Bootloader | End firmware transfer |
| DOWNLOAD_END_RESPONSE | `0x86` | Bootloader → Tool | Confirm final checksum/signature verification |
| RESET | `0x07` | Tool → Bootloader | Request MCU reset |
| RESET_RESPONSE | `0x87` | Bootloader → Tool | Confirm reset request |
| ABORT | `0x08` | Tool → Bootloader | Abort current flashing session |
| ABORT_RESPONSE | `0x88` | Bootloader → Tool | Confirm session cleanup |
| ERROR_RESPONSE | `0x7F` | Bootloader → Tool | Generic error response |

# 10. Flashing Flow

The Bootloader AI must finalize this flow first.

```text
1. Tool opens UART port.
2. Tool sends HELLO.
3. Bootloader responds with HELLO_RESPONSE.
4. Tool checks protocol version and capability flags.
5. Tool sends START_SESSION.
6. Bootloader initializes flashing session.
7. Tool sends ERASE.
8. Bootloader erases application area.
9. Tool sends DOWNLOAD_START with firmware metadata.
10. Bootloader validates firmware size, target address, checksum mode, and signature mode.
11. Tool sends DATA frames one by one.
12. Bootloader validates each DATA frame checksum.
13. Bootloader writes each block to flash.
14. Bootloader responds to every DATA frame.
15. Tool sends DOWNLOAD_END.
16. Bootloader verifies full firmware checksum.
17. If enabled, bootloader verifies firmware signature.
18. Bootloader marks application as valid.
19. Tool sends RESET.
20. Bootloader resets MCU or jumps to application.
```

---

## 10.1 Handshake Requirement

Handshake must be completed before flashing starts.

### HELLO Payload

Recommended payload:

```text
Tool Protocol Version: 1 byte
Tool Capability Flags: 4 bytes
```

### HELLO_RESPONSE Payload

Recommended payload:

```text
Bootloader Protocol Version: 1 byte
Bootloader Major Version: 1 byte
Bootloader Minor Version: 1 byte
Bootloader Patch Version: 1 byte
Bootloader Capability Flags: 4 bytes
Max Payload Size: 2 bytes
Application Start Address: 4 bytes
Application Max Size: 4 bytes
```

---

## 10.2 Firmware Metadata

The `DOWNLOAD_START` command must include firmware metadata.

Recommended payload:

```text
Firmware Size: 4 bytes
Firmware CRC32: 4 bytes
Target Address: 4 bytes
Signature Enabled: 1 byte
Signature Length: 2 bytes
Signature Data: N bytes, optional
```

---

## 10.3 DATA Frame Payload

Recommended payload:

```text
Block Index: 4 bytes
Target Offset: 4 bytes
Data Length: 2 bytes
Data: N bytes
```

---

## 10.4 Timeout and Recovery Requirement

The bootloader must implement communication timeout handling.

If timeout occurs:

1. Current command processing must be stopped.
2. The active communication state must be cleared.
3. The bootloader must keep the previous valid application untouched if possible.
4. The bootloader must return to an idle or ready state.
5. The bootloader must be able to accept a new HELLO command.
6. The tool must be able to restart the flashing flow from the beginning.

---

## 10.5 Abort Requirement

The tool must be able to send `ABORT`.

When the bootloader receives `ABORT`:

1. Stop the current flashing process.
2. Clear session state.
3. Clear temporary metadata.
4. Do not mark application as valid.
5. Return to ready state.
6. Respond with `ABORT_RESPONSE`.

# 11. Security Requirements

## 11.1 Checksum

The bootloader must support checksum validation at two levels:

1. **Frame checksum**
   - Validate each communication frame.
   - Reject corrupted frames.

2. **Firmware checksum**
   - Validate the full firmware image after transfer.
   - Do not mark the application as valid if checksum fails.

Recommended:

```text
Frame checksum: CRC16
Firmware checksum: CRC32
```

---

## 11.2 Optional Signature Verification

Signature verification must be optional and controlled by macro:

```c
#define BL_ENABLE_SIGNATURE_VERIFY 1u
```

When disabled:

- Bootloader skips signature verification.
- Tool may still generate checksum-only firmware packages.

When enabled:

- Bootloader verifies the firmware signature before marking the application as valid.
- Public key or verification material must be stored in a secure or configurable location.
- The signature algorithm must be documented.

The Bootloader AI must not implement fake security. If signature verification is not fully implemented, it must clearly mark it as a stub or future extension.

---

# 12. Application Validation and Jump

The bootloader must verify the application before jumping.

Minimum validation:

- Application start address is inside valid flash range.
- Initial stack pointer appears valid.
- Reset handler address appears valid.
- Application metadata indicates valid firmware.
- Firmware checksum has passed.
- Signature verification has passed if enabled.

The jump logic must:

1. Disable interrupts.
2. Deinitialize used peripherals if needed.
3. Set vector table to application start address if required by MCU.
4. Set stack pointer.
5. Jump to reset handler.

---

# 13. Logging Requirement

Bootloader logging must be configurable.

Supported options:

Logging must be lightweight and removable in release builds.

# 26. AI Session Handoff Requirement

Because AI coding sessions have limited context length and token capacity, both the Bootloader AI and the Flashing Tool AI must maintain a dedicated session handoff file.

This file is required so that a future AI session can continue the work smoothly without rereading or rediscovering the entire project from scratch.

The handoff file must be updated at the end of every meaningful work session and before the current AI session ends.
